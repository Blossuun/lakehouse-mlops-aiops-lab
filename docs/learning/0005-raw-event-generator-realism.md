# 0005 - 현업 느낌의 Raw 이벤트 생성기 설계 (세션/지연/중복/더티/스키마 진화)

## 배경

PR #3에서는 Raw 이벤트를 JSON Lines(.jsonl) 형태로 생성하고,
MinIO(S3 호환)에 업로드하는 흐름을 구축합니다.

목표는 단순히 랜덤 데이터를 만드는 것이 아니라,
**현업에서 실제로 겪는 문제(지연, 중복, 결측, 타입 드리프트, 스키마 변화)**를 일부러 포함하여
후속 PR(Spark/Iceberg/dbt/품질검증)에서 의미 있는 작업이 생기게 만드는 것입니다.

---

## 이번에 가졌던 궁금증과 해소

### Q1. `timeutil.py`의 역할은 무엇인가?

**요약:** “시간 규칙을 한 곳에서 강제”하기 위한 유틸입니다.

데이터 파이프라인에서 시간 관련 버그는 매우 흔합니다.
- 로컬 타임존(Asia/Seoul)과 서버 타임존(UTC) 혼용
- timezone-aware / naive datetime 혼용
- 문자열 포맷 불일치로 파싱 실패

`timeutil.py`는 다음 규칙을 강제합니다.
- UTC 기준 timestamp
- ISO8601 + `Z` 접미사

그래서 Raw 이벤트의 `event_time`, `ingest_time`을 일관되게 만들고,
파티션/지연 도착 처리의 기반을 마련합니다.

---

### Q2. `from __future__ import annotations`는 무엇인가?

**요약:** 타입 힌트(annotation)를 “지연 평가”하도록 만드는 옵션입니다.

효과:
- 타입 힌트에서 자기 자신/아직 선언되지 않은 타입을 참조할 때 더 안전해짐
- 타입 힌트로 인한 런타임 import/순환참조 문제를 줄이는 데 도움
- 점점 코드가 커질수록 타입 힌트 관리가 쉬워짐

즉, “지금은 작지만 커질 프로젝트”에서 실무적으로 자주 쓰는 패턴입니다.

---

### Q3. `dataclass`는 무엇이고, 왜 설정 객체에 사용했나?

**요약:** “데이터만 담는 클래스”를 빠르고 안전하게 만들기 위한 표준 기능입니다.

S3 설정은 항상 같은 구조를 가집니다.
- endpoint_url
- access_key_id
- secret_access_key
- region

이를 dict로 처리하면:
- 키 오타가 런타임까지 숨어들고
- 리팩토링이 어렵습니다.

dataclass는:
- 필드가 명확해지고
- 자동 생성된 `__init__`으로 실수를 줄이며
- IDE/타입 검사 도움을 받아 유지보수가 쉬워집니다.

---

### Q4. `@dataclass(frozen=True)`에서 frozen은 무엇인가?

**요약:** 인스턴스를 “불변(immutable)처럼” 만들겠다는 뜻입니다.

설정 객체는 중간에 바뀌면 디버깅이 매우 어렵습니다.
- 누가 언제 값을 바꿨는지 추적이 힘듦
- 병렬/동시 실행에서 특히 위험

그래서 설정 객체는 “한 번 만들면 바뀌지 않게” 두는 것이 실무적으로 안전합니다.

---

### Q5. `@staticmethod`는 무엇이고, `from_env` 패턴은 무엇인가?

**@staticmethod 요약:** 인스턴스(self) 없이 호출 가능한 클래스 내부 함수입니다.

`S3Config.from_env()`는 **팩토리 메서드(Factory Method)** 패턴입니다.

의미:
- 환경변수에서 설정을 읽고
- 설정 객체(S3Config)를 생성해 반환

왜 좋은가?
- env 파싱/검증 로직이 한 파일에 모임
- 다른 코드가 여기저기서 `os.environ`을 직접 읽지 않음
- 로컬/도커/CI/쿠버네티스 등 실행 환경이 달라도 “주입 방식”만 바꾸면 됨

즉, “코드와 실행 환경을 분리”하는 실무 패턴입니다.

---

### Q6. Faker 라이브러리는 무엇인가?

**요약:** 현실적인 가짜 데이터(fake data)를 생성하는 라이브러리입니다.

예:
- user agent 문자열
- 단어/문장
- 지명/주소(가짜)

단순 랜덤 문자열보다:
- 로그가 훨씬 현실적으로 보이고
- 파싱/정제/텍스트 처리 같은 작업이 의미 있어지며
- 실제 개인정보(PII)를 쓰지 않고도 유사한 형태를 만들 수 있습니다.

---

### Q7. 왜 JSONL을 Raw 포맷으로 선택했는가?

JSONL은:

- 스키마가 가변적이어도 저장 가능
- schema_version 혼합 허용
- 빠르게 생성/적재 가능

Raw 레이어는 쿼리 최적화가 목적이 아니라,
**원본 보존과 유연성 확보**가 목적이다.

---

## “현업 느낌”을 만들기 위해 의도적으로 넣은 요소들

- 이벤트 타입 분포 왜곡: view/search가 대부분, purchase는 매우 적음
- 세션/유저 단위 흐름: session_id 기반으로 묶을 수 있음
- 지연 도착: event_time < ingest_time 케이스 일부 포함
- 중복 재전송: 동일 event_id가 2회 이상 등장 가능
- 더티 데이터: 결측/타입 드리프트 일부 포함
- 스키마 진화: schema_version v1/v2를 섞어서 생성
- 참조 무결성 깨짐: catalog에 없는 product_id 일부 포함

이렇게 해야 downstream에서:
- dedup
- late data 처리
- 스키마 변화 대응
- 데이터 품질 테스트
같은 “실제 업무”가 생깁니다.

---

## 이번에 배운 점(요약)

- 현실적인 raw 데이터는 “랜덤”이 아니라 “현업 실패 모드”를 포함해야 한다.
- 시간(UTC/포맷 통일)은 파이프라인 안정성의 핵심이다.
- 설정은 코드가 읽는 것이 아니라 환경이 주입해야 한다(.env 직접 읽기 지양).
- 설정 객체를 불변으로 두면 디버깅 비용을 줄일 수 있다.
- Faker는 현실적인 텍스트/문자열을 만들 때 매우 유용하다.
